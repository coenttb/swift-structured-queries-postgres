# swift-structured-queries-postgres

⚠️ **Experimental PostgreSQL Fork** ⚠️

An experimental fork of [swift-structured-queries](https://github.com/pointfreeco/swift-structured-queries) that generates PostgreSQL-native SQL instead of SQLite.

## Why This Fork?

The original swift-structured-queries is an excellent type-safe SQL query builder for Swift, but it was designed primarily with SQLite in mind. When we attempted to build a PostgreSQL adapter layer on top of it (the previous swift-structured-queries-postgres), we encountered fundamental incompatibilities that made the layering approach impractical.

### The Failed Layering Approach

Our initial attempt was to create an adapter package that would:
1. Take the SQLite-flavored SQL generated by swift-structured-queries
2. Rewrite it at runtime to be PostgreSQL-compatible
3. Handle PostgreSQL-specific features and quirks

This approach failed due to:

- **SQL Dialect Incompatibilities**: SQLite and PostgreSQL have significant syntax differences (e.g., `INSERT OR IGNORE` vs `ON CONFLICT DO NOTHING`)
- **SERIAL Column Handling**: PostgreSQL's SERIAL columns reject NULL values, requiring complex INSERT statement rewriting
- **Runtime Performance**: String manipulation for every query added unnecessary overhead
- **Maintenance Complexity**: Every SQLite quirk needed a PostgreSQL workaround
- **Feature Limitations**: PostgreSQL-specific features like JSONB, arrays, and advanced indexes couldn't be properly expressed

### The Fork Solution

Instead of fighting these incompatibilities at runtime, this experimental fork modifies swift-structured-queries itself to generate PostgreSQL-native SQL from the start. This eliminates the need for SQL rewriting and enables proper PostgreSQL feature support.

## Key Changes from Original

This fork maintains the same API surface as the original swift-structured-queries but generates PostgreSQL-compatible SQL:

- **SERIAL Column Support**: Proper handling of PostgreSQL SERIAL columns in INSERT statements
- **CONFLICT Clauses**: Uses PostgreSQL's `ON CONFLICT` instead of SQLite's `INSERT OR` syntax
- **Type Mapping**: PostgreSQL-specific type mappings (e.g., `TEXT` instead of SQLite's flexible typing)
- **JSONB Support**: Native PostgreSQL JSONB column support
- **Boolean Literals**: Uses PostgreSQL's `TRUE/FALSE` instead of SQLite's `1/0`
- **String Escaping**: PostgreSQL-compliant string literal escaping

## Current Status

⚠️ **This is experimental software** ⚠️

- The fork is functional and used in production with [swift-records](https://github.com/coenttb/swift-records)
- Not all SQLite features have been adapted to PostgreSQL equivalents
- The test suite has been partially adapted but coverage is incomplete
- Breaking changes may occur as we discover edge cases

## Usage

This fork is designed to work seamlessly with swift-records, which provides the high-level database abstraction layer:

```swift
// In your Package.swift
dependencies: [
    .package(url: "https://github.com/coenttb/swift-structured-queries-postgres", branch: "main"),
    .package(url: "https://github.com/coenttb/swift-records", branch: "main")
]
```

## Future Direction

The goal is to eventually contribute these PostgreSQL adaptations back to the original project as a configurable SQL dialect. This would allow swift-structured-queries to support multiple database backends while maintaining a single codebase.
